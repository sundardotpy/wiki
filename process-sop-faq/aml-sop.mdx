## **What is an AML block?**

When a customer completes SEBI KYC, they are screened against our watchlists. If the screening tool returns a Probable Hit or Confirmed Hit, the customer’s account is automatically blocked pending verification, this is the “AML block.” It stays in place until CX either clears it as a false positive (with the right declaration/documents, and compliance sign-off where required) or confirms it as a true match and escalates. This SOP covers resolving AML blocks that turn out to be false positives, end to end.


# **1. Key Terms**

| **Term** | **Meaning** |
|------|---------|
| Screening report | The PDF generated by the AML screening tool (branded Screenzaa / TrackWizz) each time a customer is checked against watchlists. |
| Confirmed Hit | The screening tool found a high-confidence match on a hard identifier (typically PAN). Treat with more caution; approval is mandatory in most categories. |
| Probable Hit | The screening tool found a partial or lower-confidence match (typically on name only). Needs verification before it can be ruled a false positive. |
| Primary Match | The specific field that triggered the alert, shown in the report’s “Hit Details” table as PAN or Name. |
| AOF  | Account Opening Form, the customer’s original KYC photo and details on file, used to visually match against any photo the customer submits. |
| SEBI AML Notification system | The internal channel where the alert status is updated and the user account is unblocked once the case is closed. |

# **4. Step-by-Step Procedure**

## **Step 1: Monitor the alert**

Watch the #sebi-aml-notification Slack channel. The Slack Notifier Bot posts a message for every new hit, for example:

*“Screening Confirmed. We have got a new alert in Initial Screening. User ID: 2439277. Name of user: ALI MOHAMMAD. Type of match: Probable Hit.”*

Note the User ID, Name, and Match Type (Probable / Confirmed). You’ll need these in every later step.

## **Step 2: Request the screening report**

Reply in the same Slack thread and tag @apoorv, asking him to share the full screening report PDF for that User.

## **Step 3: Read and classify the report**

Once Apoorv shares the PDF, open it and check three fields:


1. Match Type, shown at the top as Confirm Hit or Probable Hit.
2. Primary Match, shown in the “Hit Details” table as PAN or Name.
3. Source / Watchlist: the list that flagged it, either SEBI and SAT, FCRA Orders, MHA_Designated, UNSC Consolidated List, or Interpol Wanted Persons.

Use this to place the case into one of four categories:

* Primary Match = PAN: Category C (PAN matched), regardless of source.
* Primary Match = Name, Source = MHA_Designated / UNSC / Interpol: Category D (Terror activity / Khalistani affiliation).
* Primary Match = Name, Source = SEBI and SAT / FCRA Orders, and only the first name matches in the “Match Details” table: Category A.
* Primary Match = Name, Source = SEBI and SAT / FCRA Orders, and the full name matches: Category B.\n

 If the category is not obvious from the report, ask your team lead or Vedant before proceeding. Don’t guess.

## **Step 4: Identify the documents you need**

Look up your category and hit type in the Approval Matrix (Section 5). It tells you exactly what to ask the customer for and whether compliance sign-off is mandatory or optional.

## **Step 5: Contact the User**

Call the customer first to explain why an additional declaration is needed, then follow up with the document-request email. Use the templates in Section 7, filling in the customer’s name, PAN, and the specific documents from Step 4.

## **Step 6: Verify what comes back**

When the customer replies, check that:

* The declaration is signed and follows the required wording.
* Any photo submitted visually matches the photo on the AOF.
* The DOB / PAN on the ID proof matches the KYC record.
* For PAN-match cases, the settlement/penalty receipt is genuinely from SEBI and for the correct case and amount.

**Caution:** If, after review, you believe this genuinely is the same person as the watchlist entry (a true match), stop this SOP. Escalate immediately to Vedant Somani. True matches trigger a separate regulatory process (freezing funds/assets and filing an STR with FIU-IND under the WMD Act), not a self-declaration unblock.

## **Step 7: Check if compliance sign-off is required**

Refer to the “Compliance Approval” column in the matrix (Section 5):

* Mandatory: PAN-match cases (Category C, both hit types) and Confirmed Hits on terror/Khalistani lists (Category D).
* Optional, only if you’re unsure: first-name-only matches, full-name matches, and Probable Hits on terror/Khalistani lists.

## **Step 8: Get the compliance go-ahead**

Where sign-off is required (or you’re unsure), post the case summary and the customer’s documents in #cx-legal-compliance, tagging Vedant Somani. Wait for his explicit go-ahead before moving to the next step. Never self-approve a mandatory case.

## **Step 9: Unblock the customer**

Once you’re satisfied this is not a true match (with compliance sign-off where needed), go back to the #sebi-aml-notification thread, tag @Apoorv, share the resolution, and ask him to unblock the AML flag for that User ID. Update the alert status in the thread.

## **Step 10: Close the loop**

Update the ticket with the full trail: the screening PDF, the customer’s declaration and ID proof, the compliance approval (screenshot or thread link), and your closing note. This is the audit record if SEBI or another regulator asks for it later.

# **6. How to Read the Screening Report PDF**

Every report follows the same layout, regardless of the case:

* Page 1: Input Fields (what was screened: Name, PAN, DOB, City, Country, Gender) and a Watchlist Hits summary (Total / Confirm / Probable hits) with a Hit Details table (Source, Watchlist Name, Primary Match, Match Type).
* Page 2: One block per hit, showing a Match Details table that compares the KYC value against the Watchlist value field by field (Name, PAN, DOB, Address, Gender), plus Order Details describing the underlying regulatory case.
* Page 2-3: Additional Information such as known aliases, and, for terror-list hits, a “Suggested Next Steps” section if the case were ever concluded a true match.
* Last page: Screening Profile Details, listing all 5 sources checked: UNSC Consolidated List, MHA_Designated, Interpol Wanted Persons, SEBI and SAT, FCRA Orders.

A fully matching Name, PAN, and DOB across the Match Details table is a strong signal. A match on name alone, especially a common first name, is usually the weaker, more likely false-positive case. Always verify per the matrix rather than assuming.

# **9. Common Mistakes to Avoid**

* Skipping straight to unblock without checking whether compliance sign-off is mandatory for that category.
* Accepting an unsigned declaration or one that doesn’t follow the required wording.
* Not matching the submitted photo against the AOF photo before accepting it as proof.
* Treating every Probable Hit as automatically a false positive. Always verify against the matrix and the documents.
* Continuing the self-declaration unblock flow after the documents suggest a genuine (true) match. Escalate instead, per the Caution note in Step 6.
* Closing the ticket without attaching the screening PDF, the customer’s documents, and the compliance approval trail.
